# 5.4 最小端到端闭环编排器 — 设计稿

## 1. 任务目标

将已完成模块串联为 MVP 最小可用闭环，验证以下端到端路径可以正常运行：

```
手动参考文本 → 清洗 → 切分 → 风格分析 → 用户需求 → 初稿生成 → 本地改写一个片段 → 保存输出
```

闭环完成后，用户可在项目目录中找到：风格分析报告、初稿、本地改写稿、运行日志摘要。

## 2. MVP 范围

| 步骤 | 模块 | 输入 | 输出 |
|------|------|------|------|
| 1 | `clean_text`（Task D） | 用户粘贴的原始参考文本 | 清洗后文本 |
| 2 | `split_text`（Task A） | 清洗后文本 | `list[TextChunk]` |
| 3 | `analyze_style`（Task K） | chunks + `FakeModelClient` | `StyleAnalysisResult` |
| 4 | `StoryRequirement.from_text`（Task B） | 用户自由文本需求 | `StoryRequirement` |
| 5 | `generate_draft`（Task L） | style_report + requirement + `FakeModelClient` | `DraftGenerationResult` |
| 6 | 本地改写（Task 5.3，待完成） | draft 片段 + 改写要求 + `FakeModelClient` | 改写后片段 |
| 7 | 保存 | 所有中间产物 | 项目目录文件 |

**仅完成一个最小闭环**：一次参考文本输入 → 一次风格分析 → 一次初稿生成 → 一个片段改写 → 全部落盘。不处理多章节、不处理多轮改写、不处理版本对比。

## 3. 非目标

- **不实现**多章节批量改写
- **不实现**版本对比或回滚
- **不实现**URL 抓取（只支持手动粘贴文本）
- **不实现**WebUI 或 CLI
- **不实现**多轮迭代生成
- **不实现**角色一致性自动检查（Task 5.6 的范围）
- **不引入** SQLite
- **不调用**真实 DeepSeek / Ollama（测试使用 FakeModelClient / httpx.MockTransport）

## 6. 输出结果

### PipelineResult（草案）

```python
@dataclass(frozen=True)
class PipelineResult:
    """最小闭环编排结果。"""

    # 风格分析
    style_report: str
    style_source_count: int
    style_chunk_count: int

    # 用户需求
    requirement_raw: str

    # 初稿
    draft_text: str

    # 本地改写
    rewritten_fragment: str              # 改写后的片段文本
    rewrite_original_fragment: str       # 被改写的原初稿片段（用于对比）

    # 落盘路径
    project_path: Path
    style_report_path: Path              # analysis/style_analysis_report.md
    draft_path: Path                     # drafts/draft.md
    rewritten_path: Path                 # revisions/rewritten_fragment.md

    # 摘要
    error_log: list[str]                 # 各步骤非致命错误摘要
```

### 输出文件

| 文件 | 路径 | 内容 |
|------|------|------|
| 风格分析报告 | `{project_path}/analysis/style_analysis_report.md` | 风格分析结果 |
| 初稿 | `{project_path}/drafts/draft.md` | 初稿正文 |
| 改写稿 | `{project_path}/revisions/rewritten_fragment.md` | 改写后片段 |
| 管道日志 | `{project_path}/logs/pipeline.log` | 时间戳 + 每步摘要（成功/失败/耗时） |

## 7. 流程步骤

```
┌─────────────────────────────────────────────────────────┐
│ 输入：reference_text + user_requirement_text             │
└───────────────────────┬─────────────────────────────────┘
                        │
                 ┌──────▼──────┐
                 │  1. 清洗     │ clean_text()
                 │  2. 切分     │ split_text()
                 └──────┬──────┘
                        │ chunks: list[TextChunk]
                 ┌──────▼──────┐
                 │  3. 风格分析  │ analyze_style(chunks, model_client, *, project_path)
                 └──────┬──────┘
                        │ StyleAnalysisResult
                        ├─ report → 保存到 analysis/
                 ┌──────▼──────┐
                 │  4. 需求整理  │ StoryRequirement.from_text(user_requirement_text)
                 └──────┬──────┘
                        │ StoryRequirement
                 ┌──────▼──────┐
                 │  5. 初稿生成  │ generate_draft(style_report, requirement, model_client, *, project_path)
                 └──────┬──────┘
                        │ DraftGenerationResult
                        ├─ draft_text → 保存到 drafts/
                 ┌──────▼──────┐
                 │  6. 取首个片段 │ 取 draft_text 的前 chunk_chars 字作为改写片段
                 └──────┬──────┘
                        │ fragment: str
                 ┌──────▼──────┐
                 │  7. 本地改写  │ rewrite_fragment(fragment, rewrite_instruction, local_client, *, project_path)
                 │    (待 5.3)   │
                 └──────┬──────┘
                        │ 改写后片段 → 保存到 revisions/
                 ┌──────▼──────┐
                 │  8. 写日志    │ 每步状态 + 耗时 → logs/pipeline.log
                 └──────┬──────┘
                        │
                 ┌──────▼──────┐
                 │ PipelineResult │
                 └──────────────┘
```

步骤 6 "取首个片段"规则：
- 如果初稿按章节分节（含"第X章"标记），取第一章内容
- 否则取 `draft_text` 的前 `chunk_chars` 字符，在句子边界截断（句号/问号/感叹号后）
- 片段为空时终止并返回错误

步骤 7 "本地改写"暂为占位，5.3 完成后替换为实际调用。

## 8. 错误处理策略

每步执行后检查结果。致命错误（清洗后为空、风格报告为空、初稿为空）立即终止并返回 `PipelineResult` 中携带错误摘要。非致命错误（日志写入失败）记录到 `error_log` 但不中断主流程。

所有异常使用 Task E 的 `ProjectError` / `EmptyTextError` 体系，不引入新的错误类型。

## 9. 测试方案

### 测试环境约束

- **不访问真实 DeepSeek API**
- **不访问真实 Ollama**
- **不访问网络**
- **不依赖环境变量**

### 测试分层

#### 9.1 单元测试（`tests/test_minimal_pipeline.py`）

| 测试场景 | 模型注入 | 验证点 |
|----------|----------|--------|
| 正常路径：参考文本 → 全流程 → 返回 PipelineResult | `model_client=FakeModelClient(text="风格报告")`, `local_client=FakeModelClient(text="改写片段")` | 所有字段非空；文件全部落盘 |
| 风格分析出空报告 | `FakeModelClient(text="")` | 抛出 `ProjectError(code="empty_response")` |
| 初稿生成为空 | 初稿用 `FakeModelClient(text="")` | 抛出 `ProjectError(code="empty_response")` |
| 参考文本为空 | — | 抛出 `EmptyTextError` |
| 用户需求为空 | — | 流程仍应继续（StoryRequirement 允许空） |
| 文件落盘验证 | `FakeModelClient(text="内容")` | 检查 analysis/、drafts/、revisions/、logs/ 下文件存在且内容正确 |
| FakeModelClient 记录正确请求 | `FakeModelClient(text="结果")` | 断言 `last_request.prompt` 包含参考文本片段 |

#### 9.2 FakeModelClient 使用策略（草案）

```python
def _make_fake_clients(
    style_text: str = "风格分析报告内容。",
    draft_text: str = "第一章 初稿内容...",
    rewrite_text: str = "改写后的片段内容。",
) -> tuple[FakeModelClient, FakeModelClient]:
    """同时构造云端和本地 fake 客户端。"""
    cloud = FakeModelClient(text="__sentinel__", provider="deepseek", model="deepseek-v4-flash")
    local = FakeModelClient(text="__sentinel__", provider="ollama", model="qwen3:4b")
    return cloud, local
```

> **注意**：`analyze_style` 和 `generate_draft` 共享同一个 `model_client`（云端），本地改写使用 `local_client`。FakeModelClient 的 `text` 字段需要覆盖两个不同的调用。更精确的做法：使用自定义 fake 按调用顺序返回不同文本，或注入 `httpx.MockTransport`。实现时需权衡复杂度。
