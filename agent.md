# 中文小说生成与本地改写系统需求规格说明书

## 1. 项目目标

构建一个中文小说生成与改写系统。系统通过云端强模型生成结构完整、剧情连贯的故事初稿，再通过本地部署的中文能力较强的小模型进行二次改写、角色替换、设定调整和终稿润色。

系统应长期可维护、可迁移、可测试，并适合在当前硬件环境下运行：

- GPU：RTX 3050 Laptop，约 4GB 显存
- CPU：AMD Ryzen 5 5600H
- 内存：16GB
- 首选开发环境：Windows
- 可迁移目标环境：Ubuntu / Fedora
- 主要输出语言：中文

## 2. 工程原则

本项目开发必须遵循以下四条原则。

### 2.1 先思考再编程

每次实现功能前，必须先明确：

- 当前要解决的问题是什么
- 输入是什么
- 输出是什么
- 涉及哪些模块
- 是否会影响已有功能
- 如何验证功能正确

禁止在没有理解模块边界和数据流的情况下直接写代码。

### 2.2 短代码优先

实现功能时优先选择简单、直接、可读的方案。

要求：

- 能用 100 行完成的功能，不写成 200 行
- 不提前抽象复杂框架
- 不为了“未来可能需要”创建过多层级
- 单个函数只做一件清楚的事
- 单个模块只负责一个明确职责

### 2.3 手术刀式修改

每次修改项目时，只修改完成当前任务必须修改的部分。

要求：

- 不顺手重构无关代码
- 不修改无关格式
- 不调整无关配置
- 不改变已有接口，除非当前任务明确需要
- 如果必须修改共享接口，需要同步更新调用方和测试

### 2.4 每个功能都要测试

每实现一个功能，都必须有对应测试或验证步骤。

测试优先级：

1. 单元测试：文本清洗、切分、配置读取、模型调用封装
2. 集成测试：从网址抓取到生成分析报告
3. 端到端测试：输入网址和创作要求，输出初稿和本地改写稿
4. 人工验收：检查中文流畅度、角色一致性、剧情连续性

## 3. 总体架构

```text
WebUI
  -> 后端 API
    -> 项目管理模块
    -> 小说抓取与清洗模块
    -> 文本切分模块
    -> 风格与剧情分析模块
    -> 云端初稿生成模块
    -> 本地模型改写模块
    -> 一致性检查模块
    -> 存储模块
```

系统必须将模型调用与业务逻辑解耦。云端模型和本地模型都通过统一接口调用，避免把某个模型供应商写死在业务代码里。

## 4. 推荐技术栈

### 4.1 第一版开发环境

推荐第一版使用 Windows 开发和运行。

原因：

- 更适合当前桌面使用习惯
- Ollama / LM Studio 在 Windows 上安装简单
- 方便使用 Codex 或 OpenCode 迭代
- 后期可通过配置文件和项目目录迁移到 Linux

### 4.2 后端

- Python 3.11+
- FastAPI：提供后端接口
- Pydantic：配置和数据结构校验
- SQLite：保存项目索引、生成记录、版本信息
- Markdown / JSON：保存人类可读和机器可读的项目资料

### 4.3 前端

第一版优先选择简单 WebUI：

- Streamlit：最快实现 MVP
- 或 FastAPI + 简单 HTML 页面

后续如果需要更好的交互体验，再升级为：

- React / Vue
- Open WebUI 风格界面

### 4.4 模型服务

云端强模型：

- DeepSeek API
- 默认模型：`deepseek-v4-flash`
- 用途：风格分析、剧情规划、初稿生成、结构检查

本地模型：

- Ollama 优先
- LM Studio 备选
- 本地模型用途：中文改写、角色替换、设定替换、终稿润色、局部文本重写

参考：

- DeepSeek API 文档：https://api-docs.deepseek.com/
- Ollama 文档：https://docs.ollama.com/
- Qwen3 官方仓库：https://github.com/QwenLM/Qwen3

## 5. 本地模型选择要求

由于 RTX 3050 只有约 4GB 显存，本地模型必须优先选择中文能力强、量化后可流畅运行的模型。

### 5.1 首选模型

首选 Qwen 系列小模型：

- `qwen3:4b`
- `qwen2.5:3b`
- `qwen2.5:7b` 的 Q4 量化版本

选择理由：

- 中文理解和中文输出能力较强
- 适合中文小说改写
- 小尺寸模型更适合 4GB 显存
- Ollama 支持较方便

### 5.2 模型运行策略

本地模型不负责一次性处理整篇小说，而是分段处理。

每次本地改写输入应控制为：

```text
全局设定摘要
+ 角色表
+ 当前章节摘要
+ 当前片段
+ 用户改写要求
+ 必须保留的剧情点
+ 禁止改变的事实
```

### 5.3 性能要求

最低要求：

- 本地模型可以稳定完成 1000-2000 中文字片段改写
- 单次响应不应长时间卡死
- 支持中断、重试和继续生成
- 支持降低上下文长度以换取速度

推荐参数：

- 上下文长度：2048-4096
- 量化：Q4 优先
- 温度：0.7-0.9
- top_p：0.8-0.95

## 6. 内容处理边界

系统面向虚构中文小说创作和改写。

允许目标：

- 分析小说的抽象风格特征
- 提取叙事结构、节奏、人物关系和情节模式
- 生成原创故事
- 对原创故事进行角色、设定和场景替换
- 在本地模型中进行更私密、更可控的文本改写

工程设计原则：

- 云端模型负责结构化、抽象化、高质量生成
- 本地模型负责私密化、细节化、定制化改写
- 如果云端模型拒绝某类具体表达，系统应降级为生成大纲、人物表、章节摘要，再交给本地模型继续处理

## 7. 核心功能需求

### 7.1 项目管理

用户可以创建多个小说项目。

每个项目包含：

- 项目名称
- 参考小说网址列表
- 用户创作要求
- 风格分析报告
- 剧情结构报告
- 人物关系表
- 初稿
- 本地改写稿
- 终稿
- 版本历史

项目目录建议：

```text
projects/
  project_id/
    config.yaml
    sources/
    analysis/
    drafts/
    revisions/
    final/
    logs/
```

### 7.2 参考小说抓取

输入：

- 一个或多个小说网址

输出：

- 清洗后的正文文本
- 章节结构
- 元数据

要求：

- 自动去除导航、广告、评论、脚注
- 保留章节标题
- 保留段落结构
- 抓取失败时输出明确错误
- 支持手动粘贴文本作为备选输入

### 7.3 文本切分

系统必须将长文本切分为适合模型处理的片段。

要求：

- 优先按章节切分
- 章节过长时按段落继续切分
- 不在句子中间硬切
- 每个切片保存来源信息

输出格式：

```json
{
  "chunk_id": "string",
  "source_id": "string",
  "chapter_title": "string",
  "text": "string",
  "order": 1
}
```

### 7.4 风格分析

云端模型分析参考小说的抽象风格。

输出：

- 叙事视角
- 句式特点
- 对话比例
- 描写密度
- 情绪基调
- 节奏特点
- 常见冲突类型
- 人物互动模式
- 章节推进方式

禁止输出：

- 大段原文复述
- 可直接复制的作者特征表达

### 7.5 剧情和人物分析

输出：

- 主要角色类型
- 角色关系图
- 常见剧情节点
- 冲突升级方式
- 情节转折模式
- 章节结构模板

### 7.6 用户需求输入

用户可以输入：

- 故事主题
- 世界观
- 主角设定
- 配角设定
- 人物关系
- 情节方向
- 风格偏好
- 禁止内容
- 必须出现的情节
- 结局倾向

系统应将自由文本需求整理为结构化 JSON。

### 7.7 初稿生成

云端强模型根据以下内容生成初稿：

- 风格分析报告
- 剧情结构报告
- 用户创作要求
- 角色设定
- 章节计划

输出：

- 故事大纲
- 章节大纲
- 正文初稿

要求：

- 初稿必须是原创文本
- 中文表达自然
- 角色设定一致
- 剧情有明确推进

### 7.8 本地改写

本地模型接收初稿片段和用户改写要求，输出改写版本。

要求：

- 保留核心剧情逻辑
- 根据用户要求替换角色、关系、设定或场景
- 中文表达流畅
- 避免前后人物称呼混乱
- 每段改写后输出简短修改说明

### 7.9 一致性检查

系统对改写稿进行检查。

检查项：

- 角色姓名是否一致
- 人物关系是否前后矛盾
- 时间线是否断裂
- 关键设定是否被误改
- 章节衔接是否自然
- 是否遗漏用户必须要求

## 8. 配置文件需求

系统必须使用配置文件管理模型和路径。

示例：

```yaml
app:
  language: zh-CN
  project_root: ./projects

cloud_model:
  provider: deepseek
  base_url: https://api.deepseek.com
  model: deepseek-v4-flash
  api_key_env: DEEPSEEK_API_KEY

local_model:
  provider: ollama
  base_url: http://localhost:11434
  model: qwen3:4b
  context_window: 4096

generation:
  chunk_chars: 1800
  temperature: 0.8
  top_p: 0.9
```

## 9. WebUI 需求

第一版 WebUI 页面：

1. 项目列表页
2. 创建项目页
3. 参考网址输入页
4. 抓取结果预览页
5. 风格分析页
6. 用户创作要求页
7. 初稿生成页
8. 本地改写页
9. 终稿预览页
10. 版本历史页

每个长任务都需要显示：

- 当前步骤
- 进度
- 错误信息
- 重试按钮
- 日志摘要

## 10. 错误处理需求

必须处理：

- 网页抓取失败
- 网页编码错误
- 文本为空
- API key 未配置
- 云端 API 调用失败
- 云端模型拒绝或返回空内容
- 本地模型未启动
- 本地模型超时
- 上下文过长
- JSON 解析失败

失败后系统应给出可执行建议，而不是只显示异常堆栈。

## 11. 测试计划

### 11.1 单元测试

必须测试：

- 配置读取
- URL 输入校验
- 文本清洗
- 文本切分
- JSON 结构化解析
- 模型请求封装
- 项目目录创建

### 11.2 集成测试

必须测试：

- 输入网址后生成清洗文本
- 清洗文本生成风格分析报告
- 用户需求生成结构化设定
- 初稿生成后保存到项目目录
- 本地模型改写后保存新版本

### 11.3 端到端测试

最小端到端测试：

```text
创建项目
-> 输入测试文本
-> 生成风格分析
-> 输入用户要求
-> 生成初稿
-> 本地改写
-> 输出终稿
```

### 11.4 人工验收标准

人工检查：

- 中文是否自然
- 剧情是否连贯
- 角色是否一致
- 是否满足用户要求
- 改写后是否比初稿更符合目标
- 是否存在明显重复、跑题或断裂

## 12. MVP 范围

第一版只实现最小可用闭环：

1. 创建项目
2. 手动粘贴或输入网址获取参考文本
3. 生成风格分析报告
4. 输入用户需求
5. 使用 DeepSeek 生成初稿
6. 使用 Ollama 本地模型改写一个片段
7. 保存初稿、改写稿和日志
8. 提供基础测试

暂不实现：

- 多用户系统
- 复杂权限
- 云端同步
- 自动发布
- 复杂知识库
- 大规模并发
- 精美前端

## 13. 后续迭代

第二阶段：

- 支持多章节批量改写
- 增加角色一致性检查
- 增加版本对比
- 支持导出 Markdown / DOCX

第三阶段：

- 增加更完整 WebUI
- 支持多个本地模型切换
- 支持提示词模板管理
- 支持项目级长期记忆

第四阶段：

- 迁移到 Ubuntu / Fedora
- 使用 systemd 或 Docker 管理服务
- 支持远程局域网访问

## 14. 给代码代理的实现要求

如果使用 Codex、OpenCode 或其他代码代理实现本项目，每次任务必须遵守：

1. 先阅读已有文件和当前需求
2. 先给出本次修改计划
3. 只修改完成本任务必须修改的文件
4. 每次提交尽量小
5. 每实现一个函数就补充对应测试
6. 不做无关重构
7. 不引入大型依赖，除非说明理由
8. 运行测试并报告结果
9. 如果测试无法运行，说明原因
10. 所有用户可见输出默认使用中文
